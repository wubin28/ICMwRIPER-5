  现在进入创新模式，让我们探讨几种可能的实现方案。

  方案探讨

  方案 1: 参数位置检测法

  思路： 检查第2个参数是否为
  -gitee，然后提取项目名称

  可能的参数解析逻辑：
  - 如果 $2 == "-gitee"，则 $3 是项目名称，使用
  Gitee URL
  - 如果 $2 != "-gitee"，则 $2 是项目名称，使用
  GitHub URL（默认）

  优点：
  - 逻辑直观，易于理解
  - 对现有代码改动较小
  - 命令语法清晰：icm4u create-xxx -gitee <name> vs
   icm4u create-xxx <name>

  缺点：
  - 需要修改参数数量验证逻辑（当前强制要求2个参数）
  - 两个 create 子命令都需要类似修改

  ---
  方案 2: 辅助函数提取法

  思路： 创建一个辅助函数来解析参数并返回 URL
  和项目名称

  可能包含：
  - 函数 parse_create_args() 分析参数
  - 返回：仓库类型、URL、项目名称
  - 两个 create 子命令调用同一个函数

  优点：
  - 代码复用，减少重复逻辑
  - 集中管理参数解析
  - 便于未来扩展（如果要添加其他仓库）

  缺点：
  - 增加了脚本复杂度
  - 需要引入函数定义（当前脚本是顺序执行风格）

  ---
  方案 3: 条件分支法

  思路： 在每个 create
  子命令内部，先判断参数数量，然后分支处理

  可能的结构：
  if [ $# -eq 3 ] && [ "$2" = "-gitee" ]; then
      # 使用 Gitee
      REPO_URL="https://gitee.com/..."
      PROJECT_NAME="$3"
      REPO_SOURCE="Gitee"
  elif [ $# -eq 2 ]; then
      # 使用 GitHub（默认）

  REPO_URL="https://raw.githubusercontent.com/..."
      PROJECT_NAME="$2"
      REPO_SOURCE="GitHub"
  else
      # 错误处理
  fi

  优点：
  - 在每个子命令内部自包含
  - 不改变整体脚本结构
  - 清晰的向后兼容性

  缺点：
  - 两个 create 子命令有重复的参数解析代码

  我的初步倾向

  我倾向于方案3（条件分支法）+ 输出选项B，因为：
  - 保持当前脚本的顺序执行风格

  - 不引入新的函数复杂度
  - 输出信息自然融入现有消息格式
